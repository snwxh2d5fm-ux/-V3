# AI对话记录与反馈后台 — 需求评审报告 v1.0

> 评审日期：2026-05-22  
> 评审人：分PD（生活板块）+ 营销Agent（产品运营视角）  
> 需求来源：AI质量监控页面扩展——新增对话记录浏览、质量标记、正确答案补充、调优闭环

---

## 一、需求诊断

### 1.1 现状痛点

当前AI-Chat质量运营处于"三盲"状态：

| 盲区       | 证据                                                                                                                                  |
| ---------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| 对话内容盲 | `AIQualityPage.tsx` 仅展示4个聚合数字+Top 20脱敏查询，看不到任何一条完整对话。页面自标注 `⚠️ response_preview 不在此页面展示 (P0-05)` |
| 质量判断盲 | `conversation_logs` (129条) 无任何人工质量标记字段，`eval_results` (100条) 的评分来源和维度不透明                                     |
| 反馈闭环盲 | 种子分析报告明确诊断"质量观测是盲的"为三大架构缺陷之一，`conversation_feedback` 集合在分析时甚至不存在                                |

### 1.2 需求本质

用户提出的四个功能，本质是构建一个 **Human-in-the-Loop AI质量运营体系**，属于"数据驱动×用户体验"双方法论交叉领域。

产品运营视角（营销Agent评审）：这不是一个后台功能需求，而是一个**产品质量护城河的基础设施**。在AI产品竞争进入"体验军备竞赛"阶段的当下，谁先建成人工标注→自动优化的飞轮，谁就获得不可复制的质量壁垒。

### 1.3 战略定位

将本需求放在住港伴V4的完整版图中审视：

```
住港伴三大卖点：隐私 → 合规 → 好用
                          ↑
                    本需求直接支撑"好用"
```

AI对话是用户感知"好用"的最直接触点。如果AI回复错误而无人知晓、无人纠正，"好用"就是空谈。

## 二、需求逐项评审

### 2.1 查看AI实际对话日志

**评审结论：✅ 通过（P0优先级）**

**现状**：`conversation_logs` 集合已有129条记录，`admin-ai-quality` 云函数已有 `topQueries` action做脱敏查询，但缺少单条/批量对话详情查询。

**建议实现**：

- 新增 `listConversations` action：分页列表，返回脱敏字段（会话ID、时间、用户openid前8位、问题摘要、轮次数、模型、耗时）
- 新增 `getConversationDetail` action：完整多轮对话展开，每条消息显示role/content/tokens/引用来源
- PII脱敏策略：**继承现有 `sanitize()` 函数**（手机号/证件号/邮箱正则），详情页展示前做二次脱敏。**不做原始内容透出**——这不是客服后台，是质量审核后台，脱敏后的内容足够用于质量判断。

**PD架构意见**：对话日志的查询接口应与RAG知识库查询共享同一套 `knowledge_chunks` 关联逻辑，使审核者能同时看到"AI引用了哪些知识片段"和"该片段的质量评分"，形成端到端的可追溯链。

### 2.2 标记对话质量

**评审结论：✅ 通过（P0优先级），需细化维度**

**现状**：无任何标记能力。`eval_results` 集合有 `score` 字段但无标记维度、无标记人、无标记时间。

**建议实现**：

- 标记维度从"优秀/合格/需改进/错误"四级扩展为多维评分：

| 维度   | 说明                                             | 分值 |
| ------ | ------------------------------------------------ | ---- |
| 准确性 | 回答事实是否正确                                 | 1-5  |
| 完整性 | 是否覆盖了用户问题的关键方面                     | 1-5  |
| 合规性 | 是否遵守了合规边界（无"移民"字眼、K0/K1/K2隔离） | 1-5  |
| 有用性 | 对用户的实操帮助程度                             | 1-5  |

- 综合标记：优秀(≥18分)、合格(12-17)、需改进(8-11)、错误(≤7)
- 存储：新建 `conversation_reviews` 集合，关联 `conversation_logs._id`
- 前端：每条对话卡片右侧展示评分面板，支持快速打分+备注

**营销Agent建议**：参考内容营销的"内容打分矩阵"方法论。不要让运营人员每次从头判断，而是提供**对比锚点**——同时展示AI回复、引用的RAG片段原文、以及同类问题中评分最高的回复。这降低了标注门槛，提升了标注一致性。

### 2.3 提供正确答案

**评审结论：⚠️ 有条件通过（P1优先级），需控制边界**

**价值**：这本质是在构建 **AI训练数据飞轮** 的第一环——人工标注golden answer。长期价值极高。

**风险**：

1. **边界模糊风险**：运营人员提供的"正确答案"可能引入新的合规风险（例如用了"移民"字眼）
2. **一致性风险**：不同运营人员对同一问题的"正确答案"可能矛盾
3. **覆盖幻觉**：少数几个正确答案无法覆盖8,779条知识库的盲区

**建议实现**：

- 仅对标记为"需改进"或"错误"的回复开放"补充正确答案"入口
- 正确答案提交后必须经过**二次审核**（至少另一名运营确认）才能标记为"已采纳"
- 正确答案存储为 `conversation_corrections` 集合，包含字段：原对话ID、原回复摘要、正确答案（纯文本+引用来源）、提交人、审核人、状态（待审核/已采纳/已驳回）
- 正确答案**不作为RAG直接注入**，而是进入"候选增强语料池"，由内容工程team审核后按标准流程入库

**PD架构意见**：正确答案的入库路径必须遵循数据管线规范的三道流程（清洗→校验→评估打标），不可绕过。这是数据库优先架构的底线——任何进入知识库的内容必须有结构化元数据（适用签证类型、置信度、来源、审核人）。

### 2.4 通过标记数据优化AI

**评审结论：⚠️ 方向正确，但MVP阶段不做自动优化（P2优先级）**

**原因**：

1. 种子用户仅103条对话，标记数据量不足支撑统计意义上的优化
2. AI-Chat当前架构中RAG关键词检索仍是死代码（已修复未部署），在多轮记忆、流式渲染等基础体验修复之前，优化建议可能建立在错误基线上
3. 自动优化存在"过度拟合"风险——少量标注可能导致prompt或RAG权重向个别案例倾斜

**建议实现（MVP阶段）**：

- 不做自动优化，改为**人工可读的质量报告**：
  - 按模型（Hunyuan/DeepSeek）对比评分分布
  - 按问题类型（续签/税务/证件/升学）聚类低分对话
  - 识别知识库盲区——高频出现但RAG召回为空的问题
- 提供"导出标注数据集"功能（JSONL格式），供后续prompt engineering使用

## 三、运营视角评审（营销Agent）

### 3.1 方法论映射

本需求涉及两个核心方法论：

**方法论1：数据驱动（1,422篇覆盖）**

- 应用方式：将AI质量从"凭感觉"变为"凭数据"——每条对话有评分、每类问题有画像、每个模型有对比
- 类比案例：美团外卖评价体系——不是让商家"猜测"哪里做得不好，而是用结构化评分+标签+图片让问题可视化
- 对本需求的启示：标记维度要足够细，使后续能按维度切片分析（"准确性低"和"完整性低"需要不同的修复策略）

**方法论2：用户体验（859篇覆盖）**

- 应用方式：质量标记后台的终极用户不是运营人员，而是**终端用户**——每个被标记和纠正的回复，最终受益的是下一个问同样问题的人
- 类比案例：滴滴司乘评价双向体系——乘客评分驱动司机改进，司机申诉机制防止误判。本需求中应考虑"AI申诉"——如果AI回复被标记为错误，但实际引用的是正确的政策原文，应有途径标记"非AI错误，知识库问题"
- 对本需求的启示：标记系统需要区分"AI推理错误"和"知识库内容错误"，否则会把知识库问题归咎于AI，误导优化方向

### 3.2 用户体验设计建议

**执行动作**（来自"互动参与×游戏化"战术）：

1. **质量仪表盘可视化**：不只是数字，而是趋势图+热力图+问题聚类桑基图（参考BI方案v1.1的页面行为桑基图）
2. **标注进度游戏化**：设定"本周已审核X条/目标Y条"，降低运营枯燥感
3. **快速对比模式**：同屏展示"用户问题-AI回复-RAG来源-正确答案"，一键切换

### 3.3 指标建议

| 指标             | 定义                        | 目标值（Phase 2末） |
| ---------------- | --------------------------- | ------------------- |
| 对话审核覆盖率   | 已标记对话/总对话           | ≥30%                |
| 标注一致性       | 双人标注一致率              | ≥80%                |
| 纠正采纳率       | 已采纳正确答案/提交正确答案 | ≥60%                |
| AI准确率改善     | 标记前后准确率变化          | +15pp               |
| 知识库缺口识别数 | 标注中发现的新盲区          | 20+                 |

## 四、风险矩阵

| 风险                                                 | 严重度 | 可能性 | 缓解措施                                                                                |
| ---------------------------------------------------- | :----: | :----: | --------------------------------------------------------------------------------------- |
| PII泄露——对话详情透出真实用户信息                    |  🔴高  |   中   | 云函数层二次脱敏，前端只展示脱敏后内容；管理员权限分级（仅super_admin可查看未脱敏内容） |

| 功能                                           | 排除理由                         |

## 六、架构影响评估

### 新增集合（2个）

```
conversation_reviews
├── _id: string
├── conversation_id: string (→ conversation_logs._id)
├── reviewer: string (admin_user email)
├── scores: { accuracy: int, completeness: int, compliance: int, usefulness: int }
├── overall: 'excellent' | 'good' | 'needs_improvement' | 'wrong'
├── note: string
├── reviewed_at: timestamp
└── indexes: [conversation_id, reviewer, overall, reviewed_at]

conversation_corrections
├── _id: string
├── conversation_id: string
├── review_id: string (→ conversation_reviews._id)
├── original_response_summary: string (脱敏)
├── correct_answer: string (脱敏)
├── source_refs: string[]
├── submitted_by: string
├── reviewed_by: string
├── status: 'pending' | 'approved' | 'rejected'
├── created_at: timestamp
└── indexes: [conversation_id, status, submitted_by]
```

### 新增API端点（admin-ai-quality云函数扩展）

```
新增 actions:
- listConversations(params: { page, pageSize, model?, quality?, dateFrom?, dateTo? })

- submitReview(params: { conversationId, scores, overall, note })
- submitCorrection(params: { conversationId, reviewId, correctAnswer, sourceRefs })
```

### 前端新增页面

```
src/pages/ConversationReviewPage.tsx  → 路由: /conversations
```

### 不涉及变更

- 小程序端：不涉及
- ai-chat云函数：不涉及

## 八、决策建议

**分PD（生活板块）意见**：通过。条件是正确答案补充必须遵守数据管线规范（清洗→校验→评估打标），不可绕过流程直接入库。对话日志查询应关联RAG知识片段来源，形成可追溯的质量链。

**营销Agent（产品运营）意见**：通过。建议标记维度从简单的四级标签升级为四维评分（准确性/完整性/合规性/有用性），增加对比锚点降低标注门槛，区分"AI推理错误"和"知识库内容错误"两类不同的问题根因。

**联合建议**：启动MVP开发（预计4-5天），并行修复admin-ai-quality云函数运行时异常。第一版交付后先由PM/PD试用标注20条对话，收集标注体验反馈后再决定是否扩展。
