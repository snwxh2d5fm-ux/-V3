## 二、发现清单

### DPM-01 依赖链方向错误 — NEEDS-FIX

**问题**：TDD标注 V4-005（admin-audit-trail中间件）依赖 V4-002（登录页+AuthProvider），但 V4-005 是后端云函数中间件，V4-002 是前端登录页。真实依赖是反向的：V4-002（前端）需要 V4-005（后端API）就位才能联调登录流程。

**修正依赖**：

- V4-005 依赖 → V4-003（admin_users集合已创建，中间件可查询API Key hash）
- V4-002 依赖 → V4-001 + **V4-005**（登录联调需后端鉴权接口可用）

### DPM-02 V4-007 admin-codes 严重低估 — CONCERN

**问题**：V4-007（5SP）与 V4-006（5SP）在同一个SP量级，但实际工作量相差悬殊。

| 维度         | V4-006 admin-stats |       V4-007 admin-codes       |
| :----------- | :----------------: | :----------------------------: |
| API action   |        2个         |              3个               |
| 前端组件     |        4个         |              5个               |
| 业务逻辑     |      只读聚合      | 读写+批量生成+状态管理+CSV导出 |
| 二次确认流程 |         无         |       有（密码确认弹窗）       |

**修正**：V4-007 调至 **7 SP**（或保留5SP但将前端CodeManagePage拆为独立子任务2SP）。Phase 1总SP从22调整为**24**。

### DPM-03 V4-009 P0修复合集严重低估 — CONCERN

**问题**：V4-009（3SP）打包了5项独立工作，每项都有不同的技术栈和测试要求：

| 子任务                           | 实际工作量                         | 最低SP |
| :------------------------------- | :--------------------------------- | :----: |
| content-safety-check PII自动脱敏 | 新增云函数逻辑 + 正则库 + 单元测试 | 1.5 SP |
| K2过滤集成到admin-ai-quality     | 修改云函数 + 测试验证              |  1 SP  |
| 6个新索引创建                    | CloudBase API调用 + 验证           | 0.5 SP |
| 审计日志补齐5类事件              | 修改admin-\* 5个云函数的中间件     |  1 SP  |
| page_view_logs 30天TTL逻辑       | admin-data-lifecycle新逻辑 + 测试  |  1 SP  |

**修正**：V4-009 调至 **5 SP**。Phase 1总SP从22调整为**26**。

### DPM-04 测试标准与Phase范围冲突 — NEEDS-FIX

**问题**：TF-05（兑换码CSV导出需二次密码确认）属于P1-06且对应Phase 2的V4-014任务，Phase 1无法验收。同时缺少以下测试标准：

- P0-07（6个新索引创建）缺少验证标准
- P0-01（邀请码管理就绪）缺少Phase 1验收标准
- TF-08（敏感词扫描）未覆盖admin-\*云函数返回数据

**修正**：

- TF-05 降级为P1标准，移入Phase 2验收
- 新增 **TF-01A**：6个新索引已创建且Explain Plan确认生效
- 新增 **TF-01B**：管理员可登录→生成邀请码→查看列表→统计激活率（完整流程）
- TF-08 扩展为：`grep 运营后台源码 + admin-*云函数返回字符串 → 期望0匹配`

### DPM-05 管理员CRUD管理API缺失 — NEEDS-FIX

**问题**：ADMIN_API_KEYS环境变量与admin_users集合的apiKeyHash字段存在双重存储。新增管理员需要同时写入admin_users集合+更新所有8个admin-\*云函数的环境变量，缺少自动化流程。

**修正**：V4-002或V4-005增加两个action：

```
admin-auth action: addAdmin { email, name, role }
  → 生成API Key → bcrypt hash → 写入admin_users
  → 更新ADMIN_API_KEYS环境变量 → 触发云函数重启（冷启动1-3秒）

admin-auth action: removeAdmin { email }
  → 标记admin_users.status=suspended
  → 从ADMIN_API_KEYS移除 → 更新环境变量
```

同时增加管理员列表查看和API Key轮换功能。此修正计入 V4-002 或 V4-005 的 SP 调整中（各 +1 SP）。
