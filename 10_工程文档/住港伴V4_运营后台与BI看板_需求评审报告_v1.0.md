## 二、P0问题清单（Phase 1前必须修复，共8项）

### P0-01 缺少邀请码（种子码）管理模块 【产品】

**问题**：运营后台设计了年卡兑换码管理，但缺少种子用户邀请码的管理。内测阶段的核心流程"种子码分发→激活→试用→付费转化"在运营后台完全没有工具支撑。

**影响**：运营团队无法管理内测用户。需要等Phase 3才有工具。

**修复**：Phase 1加入邀请码管理页——复用 `admin-codes` 云函数和已有 `invite_codes` 集合，通过 `codeType=invite` 筛选增加一个的 Tab 即可。预估 +0.5天。

### P0-02 admin鉴权体系与CloudBase Web Auth不兼容 【技术】

**问题**：方案中 admin-* 云函数的鉴权使用的是 `ADMIN_TOKENS` 环境变量 + 临时 token，但 CloudBase Web Auth 返回的是 **uid**（匿名 ID），与小程序 openid 不互通。运营后台用 Web Auth 登录获得的 uid 无法对应 admin_users 中的 openid。

**影响**：运营后台上线后管理员可能无法登录。

**修复**：两个方向二选一：
- A：改为 CloudBase Web Auth 的自定义声明（custom claims）做角色鉴权，admin_users 以 uid 为主键
- B：改用 HTTP 云函数 + 独立 API Key 鉴权（每个管理员分配独立 Key），配合 IP 白名单

### P0-03 page_view_logs写入量超免费额度 【技术】

**问题**：若 1,000 DAU 下 page_view + 业务事件日写入 ~65,000 次，CloudBase 免费额度 30,000 写/日，超限 2.2 倍。

**影响**：用户量增长后数据库写满或产生账单冲击。

**修复**：
- page_view 实施 1:10 采样（`sampleRate: 0.1`）
- 业务事件批量上传（每 30 秒或每 10 条合并为一次云函数写入）
- page_view_logs 设置 30 天 TTL，定时云函数清理

### P0-04 用户反馈content字段缺少PII自动脱敏 【安全】

**问题**：feedback 的 content 字段是自由文本，用户可能在描述问题时输入手机号、证件号、邮箱等 PII。当前仅 nickname 做了脱敏（"住**户"），content 完全未处理。

**影响**：PII 泄露风险。违反香港《个人资料（隐私）条例》。

**修复**：
- 反馈提交入库前，content-safety-check 云函数自动扫描手机号/证件号/邮箱模式并替换为 `[已脱敏]`
- 运营后台展示时二次过滤

### P0-05 AI对话response_preview可能进入运营后台 【安全】

**问题**：客服咨询分析看板的 AI 对话 query 聚类和词云展示，可能暴露 conversation_logs 中的 response_preview 内容——其中可能包含 K2 级政策细节或识别逻辑。

**影响**：K2 信息泄露的新途径，合规红线。

**修复**：
- response_preview 字段**绝对禁止**出现在任何运营后台页面
- 词云输入前过 K2-leak-scanner + PII 正则双重过滤
- 聚类标签使用预定义分类体系（续签/申请/证件/税务/生活），避免自动聚类生成精细化标签

### P0-06 page_view_logs缺少数据生命周期管理 【数据】

**问题**：方案完全没有提及数据归档和清理策略。三张增长表（page_view_logs、user_events、conversation_logs）无 TTL、无归档方案。

**影响**：存储膨胀、查询性能下降、账单冲击。

**修复**：
- page_view_logs：30 天 TTL，每周归档到云存储 CSV
- user_events：180 天 TTL，超期导出 JSONL 存云存储后删除
- conversation_logs：365 天 TTL
- 实现 `admin-data-lifecycle` 定时云函数（每日凌晨执行）

### P0-07 user_events + invite_codes索引缺失 【数据】

**问题**：29 种事件类型的查询需要 `{createdAt: -1}` 和 `{openid: 1, createdAt: -1}` 索引；invite_codes 扩展到双类型后需要 4 个新复合索引（codeType+status / batchId+codeType / planId+status / generatedBy+createdAt）。

**影响**：查询性能线性下降，筛选操作全表扫描，可能触发超时。

**修复**：Phase 1 部署前通过 `writeNoSqlDatabaseStructure` 创建 6 个新索引。

### P0-08 审计日志缺失关键事件 【安全】

**问题**：admin_operation_logs 仅覆盖 CRUD 操作，缺少：登录/登出事件、数据导出事件、敏感字段查看事件、权限变更事件、失败尝试事件。且审计日志本身未做 append-only 保护。

**影响**：安全事件无法追溯，不合规。

**修复**：
- 补齐 5 类审计事件
- 审计日志写入独立集合 `admin_audit_trail`，云函数层强制只 insert 不 update
- 运营后台本身无删除审计日志的权限

## 四、P2建议（Phase 3或之后处理，共5项）

1. **daily_stats_snapshots 改为增量聚合** — 技术建议，当前数据量下可以全量扫表，等用户量 >5,000 再改增量
2. **conversation_logs 增加 satisfaction / topicCategory 字段** — Phase 2 时随 AI 质量监控上线
3. **NLP 自动主题分类先人工替代** — 当前 96 条对话不足以训练分类器，Phase 2 先人工打标 50-100 条
4. **续签准备就绪度 MVP 前置到 Phase 2** — 产品建议从 Phase 3 提前，作为住港伴最核心的独家指标
5. **试用转化漏斗等精细化指标后置** — 用户量 < 1,000 时不产生有意义的信号
