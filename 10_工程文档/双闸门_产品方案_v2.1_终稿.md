## 一、概念模型（与 v2.0 一致）

| 概念 | 键 | 含义 |
|------|-----|------|
| **身份状态** | `userStatus` / `userSubStatus` | 用户在香港身份旅程中所处的阶段和角色 |
| **身份路径规划** | `selectedPath` | 用户计划通过什么路径进行身份规划 |

**核心规则：**

```
决策门禁 = isLoggedIn && userStatus 有效（非空、非 'skipped'）
```

## 二、双闸门模型（与 v2.0 一致）

```
进入应用 → 闸门1(登录?) → No→访客态L0(可浏览全部信息)
                       → Yes→闸门2(身份状态已确认?)
                              → No(skipped)→引导回status-select
                              → Yes→可做决策
```

## 四、skip-banner 与闸门 Sheet 共存方案（v2.1 新增 — P0-03）

### 4.1 问题

skipped 用户在流程控同时看到两套 UI 提示同一件事：
- skip-banner 横幅：「请先选择身份状态」+「去选择」
- 路径卡片点击 → 闸门身份确认 Sheet

### 4.2 决策

**移除 skip-banner，仅保留闸门 Sheet。** 理由：
- 一条路径比两条路径更干净，用户无需面对双重提示
- 闸门 Sheet 在用户主动发起决策时弹出，比横幅的被动展示更有上下文
- 横幅占据页面顶部空间，移除后可释放流程控的首屏展示面积

**影响：**
- `pages/process/index/index.wxml:25-32` — 删除 `skip-banner` 代码块
- `pages/documents/index/index.wxml:22-25` — 删除 `skip-banner` 代码块
- `pages/index/index.wxml:71-76` — 保留（旧首页独立页面，非 Tab）
- `pages/process/index/index.js` — 删除 `goSelectIdentity()` 方法引用（保留方法本身供旧首页使用）

### 4.3 skipped 用户在流程控的最终体验

1. 进入流程控 → 看到路径列表（信息展示，L0 浏览层）
2. 点击路径卡片 → 弹出身份确认 Sheet：「请先确认身份状态」
3. Sheet 内点击「去确认身份」→ navigateTo status-select
4. 完成身份选择 → navigateBack → 重新点击路径卡片 → 闸门通过 → 正常创建流程

## 五、路径卡片"查看"与"选择"区分（v2.1 新增 — P1-02）

当前路径卡片点击即走决策路径。为保护访客浏览体验：

**流程控的路径卡片改为两层交互：**
- 卡片主体点击 → 展开路径简介（名称、周期、适用人群、风险等级）— **无需登录**
- 简介底部按钮「选择此路径」→ 触发闸门检查 → **需要双闸门通过**

**实现方式：**
- 新增 `data.expandedPathId` 跟踪当前展开的路径卡片
- 点击卡片 → setData 展开/收起
- 展开区域内包含绑定 `onSelectDirectPath` 的「选择此路径」按钮
- 不改变现有 WXML 结构，仅在卡片内增加展开区和按钮

## 六、AI 对话门禁策略（v2.1 新增 — P0-02 + P2-01）

### 6.1 floating-ai 路径采纳门禁

`selectPathFromChat` 在赋值 `selectedPath` 之前加入 `canMakeDecision()` 检查。
方案为轻量 Toast（UI 嵌套限制），PD 确认可接受。

**Toast 文案：**
- 闸门1失败（未登录）：`「登录后可保存路径选择」` — 引导性而非禁止性
- 闸门2失败（skipped）：`「请先在「我的」补选身份状态后选择路径」`

### 6.2 AI Chat 独立页面匿名试用策略

`subpkg-chat/.../chat` 允许访客匿名发送消息（体验价值）。不强制登录。
- 第 3 条消息回复末尾追加引导文案：`「登录后可保存对话记录，享受个性化路径推荐」`
- 优先级 P2，核心闸门完成后再处理。
