# 双闸门方案 — 验收不通过标准与场景 v1.0
> 2026-05-21 | 对应产品方案 v2.1 终稿 | 发版前必检

## 一、总则

以下任一条触发即为**整体验收不通过**，不得合并至主分支或推送生产。每条标注阻断级别：

| 级别 | 含义 |
|:--:|------|
| 🔴 | 阻断发版。此条不通过 = 版本不可发布 |
| 🟡 | 严重缺陷。必须修，但不阻止内测/灰度 |
| 🟢 | 体验缺陷。正式版前需修，可随下个 hotfix |

---

## 二、闸门完整性（🔴 阻断）

### F-01：决策入口被绕过

**不通过条件：** 以下 5 个入口中任一在未登录或身份为 skipped 时仍可成功创建流程线或设置 selectedPath。

| # | 入口 | 失败信号 |
|---|------|---------|
| F-01a | `process/onSelectDirectPath` | 未登录/skipped 用户点击路径卡片后创建了流程线 |
| F-01b | `process/selectTemplate` | 未登录/skipped 用户选择模板后创建了流程线，且 `userStatus` 被覆写为 `'unapplied'` |
| F-01c | `path-select/onSelect` | 未登录/skipped 用户在独立选择页成功创建流程线 |
| F-01d | `assessment-result/selectPath` | 未登录/skipped 用户评估后成功设置 `selectedPath` |
| F-01e | `floating-ai/selectPathFromChat` | 未登录/skipped 用户采纳AI推荐后成功设置 `selectedPath` |

**验证方法：** 对每个入口，构造未登录 和 skipped 两种状态，执行入口操作，断言流程线未被创建且 selectedPath 未被写入。

### F-02：canMakeDecision() 判定逻辑错误

**不通过条件：** 以下任一状态组合的判定结果与预期不符。

| 状态 | isLoggedIn | userStatus | 预期 canMakeDecision() |
|------|:--:|------|:--:|
| 未登录 | false | null | `{ok:false, reason:'login'}` |
| 跳过登录 | false | null | `{ok:false, reason:'login'}` |
| 跳过身份 | true | 'skipped' | `{ok:false, reason:'identity'}` |
| 已确认-未申请 | true | 'unapplied' | `{ok:true}` |
| 已确认-已提交 | true | 'submitted' | `{ok:true}` |
| 已确认-已获批 | true | 'approved' | `{ok:true}` |
| 已确认-永居 | true | 'permanent' | `{ok:true}` |
| 异常-空字符串 | true | '' | `{ok:false, reason:'identity'}` |

**验证方法：** Jest 单元测试直接调 `canMakeDecision()`，mock `getApp().globalData` 和 `wx.getStorageSync`，覆盖全部 8 行。

### F-03：全局替换/搜索遗漏

**不通过条件：** 项目中仍存在任何未受闸门保护的 `selectedPath` 赋值或流程线创建调用。

**搜索范围：**
```
grep -rn 'selectedPath\s*=' --include='*.js' pages/ components/ subpkg-*/ | grep -v node_modules
grep -rn 'saveProcessLine' --include='*.js' pages/ subpkg-*/ | grep -v node_modules
```

每处赋值必须人工确认是否属于 5 个已加闸门的入口之一。发现未受保护的新入口 = 不通过。

## 三、数据完整性（🔴 阻断）

### F-04：selectTemplate 覆写 userStatus 未被阻断

**不通过条件：** skipped 用户通过 selectTemplate 创建流程后，`userStatus` 从 `'skipped'` 变为 `'unapplied'`。

**验证方法：** 构造 skipped 状态，触发 selectTemplate 入口，检查闸门阻断后 `userStatus` 仍为 `'skipped'`，`__user_status__` Storage 未被修改。

### F-05：闸门阻断后残留脏数据

**不通过条件：** 任意决策入口被闸门阻断后，Storage 或 globalData 中出现以下任一项：
- `__active_process_id__` 被写入新值
- `__selected_path__` 被写入新值
- `__process_stage__` 被修改
- 新的 processLine 被保存至本地流程线列表

**验证方法：** 对每个入口，在闸门阻断前后分别快照 Storage 键值，断言全部决策相关键值未变更。

### F-06：身份状态语义污染

**不通过条件：** 访客态（未登录）用户的 `userStatus` 被任何代码路径写入为非 null 值（包括 `'skipped'`）。

**说明：** 访客没有 `userStatus`。`skipLogin()` 不写状态——这是正确的。如果后续代码误将访客当作 skipped 处理，会导致闸门2错误触发（应该触发闸门1）。

**验证方法：** 未登录状态下遍历所有Tab和子包页面各操作一次，断言 Storage 中 `__user_status__` 始终为 undefined/null。

## 四、用户体验（🟡 严重缺陷）

### F-07：双重提示未消除

**不通过条件：** skipped 用户在流程控或证件夹页面同时看到 skip-banner 横幅和闸门 Sheet。

**验证方法：** 构造 skipped 状态，进入流程控 Tab，断言无 skip-banner 渲染。进入证件夹 Tab，断言无 skip-banner 渲染。

### F-08：登录 Sheet → 身份 Sheet 串联断裂

**不通过条件：** 未登录用户在闸门 Sheet 中完成微信登录后，若 `userStatus` 仍为 null/skipped，Sheet 消失或卡死，无后续引导。

**预期行为：** Sheet 自动从 login 模式切换至 identity 模式，用户可继续点击「去确认身份」。

**验证方法：** 构造"登录后无身份状态"场景（如新建账号），触发闸门 → 登录 → 断言 Sheet 显示 identity 模式而非消失。

### F-09：Dismiss 后状态不恢复

**不通过条件：** 用户点击闸门 Sheet 的「稍后再说」/「暂不」后，再次点击同一决策入口时 Sheet 不弹出，或弹出但状态错误。

**验证方法：** 点击路径卡片 → Sheet弹出 → 点关闭 → 再次点击同一路径卡片 → 断言 Sheet 正确弹出。

### F-10：selectTemplate gate-passed 后无合理恢复路径

**不通过条件：** 用户在 selectTemplate 被闸门阻断 → 完成身份确认 → gate-passed 回调执行后，用户不知如何继续（模板面板已关闭且无提示）。

**预期行为（二选一）：**
- 方案A：gate-passed 自动重新打开模板面板并选中之前的模板（pendingTemplateId）
- 方案B：gate-passed 后 Toast 提示「身份已确认，请重新选择路径」，用户手工重选

两种方案均可通过，但不能出现"静默关闭，用户困惑"的状态。

## 五、回归保护（🔴 阻断）

### F-11：已登录已确认身份用户零回归

**不通过条件：** 已登录且 `userStatus` 为 `'unapplied'`/`'submitted'`/`'approved'`/`'permanent'` 的用户，在以下任一操作中行为与加闸门前不一致：

| 操作 | 预期行为（不变） |
|------|----------------|
| 流程控点击路径卡片 | 正常创建流程线 |
| 流程控选择模板 | 正常创建流程线 |
| path-select 选择路径 | 正常创建流程线 |
| 评估结果页选路径 | 正常设置 selectedPath |
| AI 对话采纳推荐路径 | 正常设置 selectedPath |

**不通过的具体信号：**
- 闸门 Sheet 误弹出
- Toast "请先登录" 误显示
- 流程线创建失败
- 页面跳转异常

**验证方法：** Jest 回归测试全量通过（365/369）+ 真机走过 5 个入口的操作流程。

### F-12：现有 skip-banner 功能未因移除而被破坏

**不通过条件：** 旧首页（`pages/index/index`）的 skipped 引导卡片（hero-empty hero-skip）被误删或不可见。

**说明：** 旧首页不是 Tab 页，其 skip-banner 保留。仅移除流程控和证件夹的 skip-banner。

**验证方法：** 构造 skipped 状态，`wx.navigateTo('/pages/index/index')`，断言 hero-skip 卡片正常渲染。

### F-13：¥599 身份重置流程不受影响

**不通过条件：** 用户支付 ¥599 重置身份后，以下流程任一出现异常：
- Storage 清除不完整
- `wx.redirectTo status-select?mode=reset` 未触发
- 重置后 `userStatus` 不为 null
- 重置后再次 skipIdentity → 闸门行为异常

**验证方法：** 完整走通 ¥599 支付 → 重置 → skipIdentity → 进入流程控 → 点击路径卡片 → 闸门弹出 identity Sheet 的完整链条。

### F-14：Jest 无新增失败

**不通过条件：** 全量 Jest 测试结果与基线（14/14 suites, 365/369 pass, 4 todo）相比出现新增失败项。

**验证方法：** `npx jest --passWithNoTests --forceExit`，断言 pass 数 ≥ 365，fail 数 = 0（ai-chat mock API key 的 4 todo 除外）。

## 六、边界场景（🟡 严重缺陷）

### F-15：Storage 损坏容错

**不通过条件：** `__user_status__` 在 Storage 中为异常值（非字符串、undefined、数字）时，`canMakeDecision()` 抛出异常导致页面崩溃。

**预期：** 任何异常 Storage 值均应降级为 `reason:'identity'`，不抛异常。

**验证方法：** Jest 单元测试注入 `wx.getStorageSync` mock 返回 `null` / `123` / `undefined` / `{}`，断言函数正常返回不抛异常。

### F-16：多次快速点击闸门入口

**不通过条件：** 用户快速双击路径卡片时，闸门 Sheet 弹出两次重叠，或创建了重复的流程线。

**预期：** `showGateSheet` 已为 true 时，`setData` 不再触发新 Sheet。流程线在闸门通过前不创建。

**验证方法：** 真机快速双击路径卡片，断言仅弹出一个 Sheet，Storage 中无双份流程线。

### F-17：网络异常时登录 Sheet 不卡死

**不通过条件：** 用户在闸门 Sheet login 模式中点击登录，若云端 `user-auth` 不可达，Sheet 永久转圈不恢复。

**预期：** 8 秒超时后显示错误提示，用户可关闭 Sheet 或重试。

**验证方法：** 断网 → 触发闸门 → 点击微信登录 → 断言 8 秒内出现错误状态，Sheet 可正常关闭。
