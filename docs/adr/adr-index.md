# ADR-001: 通讯协议 v3 — MCP直连替代outbox心跳

**日期**: 2026-05-14  
**状态**: 已采纳  
**决策人**: 琅琊

## 背景

v2协议采用7步outbox/inbox断链心跳，导致Claude写完代码后需要等待Hermes轮询outbox，延迟高且容易断链。

## 决策

采用MCP直连协议：
- Claude写完代码 → 同会话调Hermes MCP verify/deploy（闭环）
- outbox仅存跨会话事项（设计方案/审查报告/ACL审批）
- PROGRESS.md单源真相，所有任务状态只在此处
- CLAUDE.md首条=启动必读PROGRESS.md

## 影响

- 减少断链风险
- 闸门执行更及时
- outbox积压减少

# ADR-002: 代码唯一来源 — Claude

**日期**: 2026-05-13  
**状态**: 已采纳  
**决策人**: 琅琊

## 背景

多个Agent（Hermes/麒麟/玄武）都有能力修改代码，导致互相覆盖、反复回退。

## 决策

代码只有一个来源：Claude。其他Agent发现Bug → 写inbox → ACL通知Claude。

违规 = 信用分 -30/次，<60 熔断。

## 影响

- 代码变更可追溯
- 避免多Agent同时修改同一文件
- Hermes专注测试/验证/部署

# ADR-003: 双审机制 — 麒麟+玄武并行

**日期**: 2026-05-15  
**状态**: 已采纳  
**决策人**: 琅琊

## 背景

静态闸门（verify.sh/Jest/DevTools）只检查代码红线，无法捕获逻辑错误和功能完整性问题。

## 决策

每次Claude交付后，Hermes必须并行委派麒麟（代码审查）和玄武（PM审查）。

## 影响

- 能捕获参数位移、类型Bug、WXML缺渲染块等静态闸门漏过的问题
- 增加约10-15分钟审查时间
- 审查报告写入inbox，作为Claude了解闸门结果的入口
