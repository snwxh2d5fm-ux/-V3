# 数据处理管线规范

> 规则类型: 硬约束 (所有入库数据必须遵循)
> 适用于: 所有批次数据处理、知识库建设、数据库写入操作

## 核心规则

### 1. 原数据与处理数据严格区隔
- 原数据锁定后不得直接引用
- 所有产品引用必须经过三道流程处理

### 2. 三道路程

#### 第一程：清洗
- PII 脱敏（姓名/电话/证件号/地址 → `[已脱敏]`）
- 营销内容过滤（邀请码/推广链接/付费广告）
- 格式标准化（统一日期格式/币种标注/路径分类）
- 去重（contentHash 匹配，相同内容仅保留一条）

#### 第二程：校验
- 事实核查（政策条文必须可追溯到官方来源）
- 跨源一致性（同一问题多个来源答案矛盾时标记为🟡）
- 时效性（政策信息标注最后确认日期，超过12个月标记为待复审）
- 追溯链（每条数据必须标注来源、采集时间、清洗时间）

#### 第三程：评估打标
- 绿/黄/红等级（绿色可直接引用，黄色需加来源标注，红色禁止引用）
- 置信度（五级：确定/高/中/低/不确定）
- 来源可靠性（A官方/B机构/C社区/D个人）
- 知识域标签（K0/K1/K2）

### 3. 入库协议
- 统一 JSONL 格式，每行一条记录
- 必填字段: `_id`, `contentHash`, `source`, `sourceType`, `knowledgeDomain`, `confidence`, `grade`, `processedAt`
- K2 级数据不得入知识库（仅供内部参考）

### 4. 禁止事项
- 禁止跨库引用（产品引用只能来自处理后的知识库）
- 禁止引入夸大、虚假、营销内容
- 禁止将 K2 级数据写入 knowledge_chunks
- 杭州入园等跨域语料仅萃取方法论，不入库实务信息

## 批次命名规范
- 格式: `EX-YYYY-MMDD-NNN`
- 示例: `EX-2026-0507-001`
- 每批次需有完整的 raw → processed → audit 三文件
