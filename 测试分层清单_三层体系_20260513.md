## 分层定义

| 层级 | 执行方式 | 触发频率 | 单次耗时 | 覆盖率 |
|:----:|---------|---------|:------:|:-----:|
| **L1** | `npm run test:e2e` (automator) | 每次 push | 5-10 min | 72/90 (80%) |
| **L2** | WeTest 云真机脚本 | 每日/PR合并 | 15-30 min | 6/90 (7%) |
| **L3** | 人工真机 | 发版前 / 高风险变更 | 30-60 min | 12/90 (13%) |

---

## 分类规则

一条测试项归入 L1，当且仅当它的**全部操作和断言**都可以在开发者工具模拟器中完成：

- ✅ 页面跳转、元素存在性、文本内容、Tab 切换
- ✅ 表单输入（文本类）、数据读写（storage / 云函数调用）
- ✅ 弹窗行为、状态切换、DOM 属性断言
- ❌ 不涉及相机、支付、微信登录授权、系统级操作、屏幕阅读器

归入 L2 当它的操作需要真实设备但**可以脚本化**：暗色模式切换、网络断开、多机型渲染差异、大文件处理。

归入 L3 当必须**人工判断或系统级交互**：相机拍照、OCR 识别、VoiceOver 行为、真实微信登录、杀进程恢复。

## L2 — WeTest 云真机 (6项)

需要真实设备环境，但可以通过 WeTest 脚本自动化，不占人力。  
生成脚本: `npm run wetest:gen`

| # | 测试项 | 分类理由 | 关键验证点 |
|:-:|------|------|------|
| 0.7 | 令牌系统-无硬编码色 | automator 可取 computed style，但颜色在真机屏幕上的**实际呈现**需要人眼确认或 WeTest 截图对比 | 多机型色差不超过 ΔE 3.0 |
| 0.8 | 令牌系统-暗色/亮色 | automator 无法切换**系统级**暗色模式，真机/WeTest 可以 | iOS Dark Mode + Android Dark Theme 均适配 |
| 0.13 | 网络状态监听 → 离线横幅 | automator 无法在运行时断开网络连接 | 断网后小程序不崩溃、恢复网络后状态正确 |
| 1.1 | 冷启动 → <3秒无白屏 | automator 的 reLaunch 不是真实冷启动，WeTest 可测量从点击图标到首页渲染的完整耗时 | 启动耗时 <3s (iOS) / <4s (Android) |
| 10.1 | 无网络 → 离线横幅不崩溃 | 同 0.13，真机可控网络环境 | 各 Tab 离线状态一致 |
| 10.3 | 大图上传 → >10MB 提示不崩溃 | automator 的 file system 与真机不同，大文件处理行为有差异 | 提示文案正确 + 不闪退 |

## L3 — 人工真机 (12项)

无法用 automator 或 WeTest 脚本替代，必须由人手持设备操作和判断。

### 相机 / OCR 链路 (4项)

| # | 测试项 | 不可自动化原因 |
|:-:|------|------|
| 3.2 | 添加证件-拍照 → 相机→OCR | `wx.chooseImage` / `wx.scanCode` 是原生 API，automator 无法触发真实相机 |
| 4.2 | 添加提醒-OCR → 拍照→识别日期 | 同上，OCR 识别结果需要在真机上用真实证件照片验证 |
| 11.1 | 拍照质量检测 → 6项评分 | 需要真实拍摄不同质量的照片（反光/模糊/倾斜），验证评分准确性 |
| 11.2 | 拍照无OCR → 直接进入手动填写 | 相机拍照后的**流程分支**需要在真机验证（模拟器行为不一致） |

### 微信登录授权 (2项)

| # | 测试项 | 不可自动化原因 |
|:-:|------|------|
| 1.4 | 新用户登录 → 跳转身份选择 | `wx.login` 需要真实微信账号的 code 换取 openid，模拟器无法完成 OAuth 全链路 |
| 1.5 | 回访用户登录 → status-badge 显示 | 依赖真实 openid 关联的数据库记录，mock 数据与真实链路行为有差异 |

### 无障碍 (3项)

| # | 测试项 | 不可自动化原因 |
|:-:|------|------|
| 0.9 | VoiceOver → 图片有文字描述 | 屏幕阅读器的朗读行为只有人耳能判断"是否自然、是否完整、是否按正确顺序" |
| 0.10 | VoiceOver → 弹窗类型+标题 | aria 属性可以在代码层验证存在（L1补充），但 VoiceOver 实际朗读效果需人工 |
| 3.10 | VoiceOver → 证件列表 aria-label | 同上，代码可验证属性存在（L1补充），朗读效果需人工 |
| 7.7 | VoiceOver → floating-ai 按钮描述 | 同上 |

> **L1 补充策略**: 对无障碍项，automator 可做 **属性存在性检查** (aria-label 非空、aria-modal=true)，作为 L3 的前置过滤。属性缺失 → L1 直接拦截，属性存在 → 仍需 L3 验证朗读质量。

### 系统级操作 (2项)

| # | 测试项 | 不可自动化原因 |
|:-:|------|------|
| 9.1 | 杀进程恢复 → 数据不丢失 | 微信小程序的杀进程行为由操作系统管理，automator 无法模拟 |
| 9.2 | 隔天恢复 → 提醒到期状态更新 | 依赖系统时钟变化和服务端定时任务，automator 无法快进时间 |
