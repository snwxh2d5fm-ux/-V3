## 🔴 P0 — 阻断 (Phase 1启动前必须修复)

### PRD对齐维度

#### P0-01: 方案设计v1.1路线图与需求评审修正路线图冲突 [路线图断裂]

- **问题**: 方案设计§九 的三阶段路线图（原版）总计28天，需求评审报告§五 输出修正路线图总计28天。但两份路线图的Phase 1内容不同：
  - 方案设计v1.1 Phase 1: 基础设施建设 + 全量34页埋点 (12天)
  - 需求评审修正Phase 1: 种子期运营工具 + 核心15页埋点 (8天)
- **偏差**: 两个版本的Phase 1任务清单、工时、交付物完全不同。开发团队不知道该按哪个版本执行。
- **影响**: Phase 1启动时必然产生返工和排期混乱。
- **修复**: 明确唯一路线图版本。建议以需求评审修正版为准（已吸收8项P0修复 + 路线图节奏优化），更新方案设计文档或标记原路线图为deprecated。

#### P0-02: admin鉴权方案与需求评审P0-02修复方案未收敛 [PRD未更新]

- **问题**: 方案设计§6.3 定义鉴权方式为 "ADMIN_TOKENS环境变量 + 临时token + admin_users集合白名单"。需求评审P0-02指出CloudBase Web Auth返回uid与小程序openid不互通，修复方案为二选一（自定义声明 or HTTP云函数+API Key）。
- **偏差**: 方案设计未反映该P0修复决策。若按原方案实施，登录体系上线后不可用。
- **影响**: 运营后台登录功能无法正常工作，Phase 1交付物不可验收。
- **修复**: 确认鉴权方案（推荐方案B: HTTP云函数+独立API Key+IP白名单），更新方案设计§6.3和§7.1 admin_users集合的openid→uid主键变更。

#### P0-03: 邀请码管理模块在方案设计中缺失种子码管理 [PRD遗漏]

- **问题**: 方案设计§3.4.5 定义了年卡兑换码管理（codeType=redemption），但完全未覆盖种子用户邀请码（codeType=invite）的管理页面。需求评审P0-01明确指出此缺失。
- **偏差**: 运营后台无法管理内测阶段的种子用户。`invite_codes`集合已有数据但无前端管理界面。
- **影响**: 内测期间运营团队无工具分发/追踪种子码激活，需依赖数据库直操作。
- **修复**: 方案设计补充 §3.4.7 邀请码管理模块，与兑换码管理共用 `admin-codes` 云函数和 `invite_codes` 集合，通过 `codeType` Tab切换。

### 数据流闭合维度

#### P0-04: page_view → daily_stats_snapshots 聚合链路无定义 [数据流断裂]

- **问题**: 方案设计§5.2 定义了34页面page_view埋点写入 `page_view_logs`，§7.1 定义了 `daily_stats_snapshots` 的 pageViews 字段按模块聚合。但：
  1. 未定义从 `page_view_logs` 到 `daily_stats_snapshots.pageViews` 的聚合逻辑
  2. 未定义聚合由哪个云函数执行（admin-stats? 独立定时函数?）
  3. 未定义page→module的映射表（哪些页面归入guidebook/documents/reminders/process/ai_chat/mine）
- **影响**: BI看板首页的页面热度模块无数据源，`daily_stats_snapshots.pageViews`永远是空对象。
- **修复**:
  1. 补充page→module映射表（例: subpkg-guide/pages/\* → guidebook）
  2. 在 `admin-stats` 或新增 `admin-data-lifecycle` 云函数中定义日终聚合逻辑
  3. 方案设计§5.2 补充页面-模块映射列

#### P0-05: user_events 29种事件 → 试用转化漏斗 查询路径不闭合 [数据流断裂]

- **问题**: 方案设计§3.4 试用转化漏斗定义为 "trial_started → process_created → document_added → reminder_created → order_paid"，附录C示例代码使用 `db.collection('user_events').where({eventType: step}).count()`。
- **偏差**:
  1. `where({eventType: step}).count()` 得到的是全局事件计数，不是同一用户群组的漏斗转化。漏斗需要按用户openid做跨事件类型关联。
  2. 29种事件类型中，方案设计列出了 `trial_started`、`order_paid`，但 `process_created` 和 `reminder_created` 在已有事件清单中（5.1节），新增的 `reminder_responded` 等未纳入漏斗步骤。
  3. 现有 `user_events` 集合仅146条记录且只有2种eventType，29种事件的数据字段定义未完成。
- **影响**: 漏斗数据不准确——显示的是各事件的总次数，不是真实的用户级转化率。
- **修复**:
  1. 漏斗查询改为：先取所有 `trial_started` 用户openid集合 → 逐一检查后续事件是否存在
  2. 补充 `user_events` 29种事件的完整字段定义（每个事件的payload结构）
  3. 明确 `process_created` / `document_added` / `reminder_created` 是否需新增为显式事件（当前仅存在旧usage-tracker中）

#### P0-06: page_view_logs / user_events / conversation_logs 三表无TTL [数据生命周期断裂]

- **问题**: 同需求评审P0-06。方案设计§十一 提到了page_view埋点写入量风险，但§七 的三个新增集合均未定义TTL。
- **额外发现**: `daily_stats_snapshots` 集合本身也无归档策略——按日增长365条/年，长期积累无上限。
- **影响**: 存储膨胀、查询性能下降、CloudBase账单冲击。
- **修复**:
  1. page_view_logs: 30天TTL + 每周CSV归档到云存储
  2. user_events: 180天TTL + 超期JSONL归档
  3. conversation_logs: 365天TTL
  4. daily_stats_snapshots: 保留全量（数据量小），但增加年度归档策略
  5. 实现 `admin-data-lifecycle` 定时云函数

#### P0-07: 续签准备就绪度评分 数据源5维度采集路径未定义 [数据流不闭合]

- **问题**: 方案设计§4.3 定义 "续签准备分 = 证件齐全度×0.3 + 居住记录完整度×0.25 + 提醒设置率×0.2 + 流程完成度×0.15 + AI评估分×0.1"。
- **偏差**:
  1. "居住记录完整度" — 当前数据库无此字段，无采集路径
  2. "证件齐全度" — 依赖documents集合，但"应添加证件数"未定义（按路径/按阶段？按什么标准？）
  3. "AI评估分" — 数据来源是 `eval_results` 还是 `conversation_logs`？
  4. 五个维度的权重总和1.0，但每个维度的归一化方法未定义（0-100如何映射？）
- **影响**: 住港伴最核心的独家指标无法计算，P0-04路线图建议将其前置到Phase 2但无数据基础。
- **修复**:
  1. 补充"居住记录完整度"的数据采集方案（需新增字段或事件）
  2. 明确"应添加证件数"的计算规则（按路径的document-index-templates总数）
  3. 明确AI评估分来源
  4. 补充归一化公式

### 安全合规维度

#### P0-08: 运营后台本身无删除审计日志的权限 —— 未在设计文档中落地 [合规缺口]

- **问题**: 需求评审P0-08指出审计日志需补齐5类事件 + append-only保护 + 运营后台无删除权限。方案设计§7.1 的 `admin_operation_logs` 仅覆盖 {action, targetType, targetId, detail} 四字段。
- **偏差**:
  1. 未定义独立的 `admin_audit_trail` 集合（与需求评审修复要求不符）
  2. 未定义登录/登出、数据导出、敏感字段查看、权限变更、失败尝试5类审计事件
  3. 未声明append-only约束
- **影响**: 安全事件无法追溯，不合规。
- **修复**:
  1. 方案设计§7.1 新增 `admin_audit_trail` 独立集合
  2. 补充5类审计事件类型定义
  3. 声明云函数层强制只insert不update
  4. 运营后台前端不提供删除审计日志的UI

## 🟡 P1 — 重要 (Phase 1-2内修复)

### PRD对齐维度

#### P1-01: 方案设计首页仪表盘11项指标 vs 需求评审精简至6项 [不一致]

- **问题**: 方案设计§3.2 用户总览含6类信息卡片，§4.1 BI一级指标含11项。需求评审P1-01建议Phase 1精简至6项核心指标，其余P2后置。
- **偏差**: 两份文档对Phase 1首页交付物定义不同。
- **修复**: 统一为6项核心指标：累计用户(按签证分层)、活跃用户、AI准确率、安全事件、月收入、兑换码激活率。其余指标标记Phase 2/3。

#### P1-02: 34页全量埋点 vs 15页核心埋点 [过度设计]

- **问题**: 方案设计§3.3.5 定义了34页面page_view埋点，但需求评审P1-02指出核心15页承载80%行为，设置/关于/发票列表/发票详情等页面无独立埋点必要。
- **偏差**: Phase 1 埋点工作量从1.5天(15页)膨胀为3天(34页)，且page_view_logs写入量翻倍。
- **修复**: Phase 1 只埋核心15页，29种事件精简至20-22种。剩余页面在Phase 2补充。

#### P1-03: 客服咨询分析看板中AI对话response_preview可能进入运营后台 [PRD安全对齐]

- **问题**: 方案设计§3.8 客服咨询分析看板提到"基于AI对话query的NLP聚类"和"脱敏后的用户query词云"，但未明确声明response_preview字段绝对禁止出现在任何运营后台页面（需求评审P0-05的修复要求）。
- **偏差**: 方案设计v1.1在v1.0基础上补充了客服看板，但未同步吸收P0-05的修复。
- **修复**: 方案设计§3.8 增加明确声明："response_preview字段绝对禁止出现在任何运营后台页面；词云输入前过K2-leak-scanner+PII正则双重过滤；聚类使用预定义分类体系。"

#### P1-04: content-safety-check未被share-create调用 [合规缺口复用]

- **问题**: 前序R2 P0-03 + R3 P0-02 标记 `share-create` 未调用 `content-safety-check`。V4运营后台方案设计中未考虑分享内容的合规审核接入。
- **影响**: 运营后台作为管理端，不应仅依赖小程序端的审核——管理端的内容运营模块也需展示经审核的分享内容。
- **修复**: 运营后台内容运营模块展示分享内容前，确认 `share-records` 中仅包含经 `content-safety-check` 审核通过的内容。

### 数据流闭合维度

#### P1-05: admin-feedback 客服效率指标 首次响应时间 数据源缺失 [数据流不闭合]

- **问题**: 方案设计§3.8 定义客服效率指标含"平均首次响应时间"，但：
  1. `feedback` 集合无客服响应时间戳字段
  2. `feedback_reply` 集合无客服身份字段（无法区分客服回复 vs 用户追加）
  3. 工单状态变更时间线未记录
- **影响**: 客服效率指标无法计算。
- **修复**:
  1. feedback集合增加 `firstReplyAt` 字段
  2. feedback_reply增加 `replierRole: user|cs` 字段
  3. 或使用 `admin_operation_logs` 记录客服操作时间

#### P1-06: 页面行为分析「流失页面识别」定义不完整 [数据流定义缺陷]

- **问题**: 方案设计§3.3.5 行为分析看板第7项"流失页面识别：高跳出率页面（访问后直接退出小程序的页面）"。
- **偏差**:
  1. 微信小程序无"页面退出"事件——用户关闭小程序时不会触发任何上报
  2. `page_leave` 事件仅在页面 `onHide` 时触发，但onHide≠退出小程序（可能切换到聊天）
  3. "跳出率"的定义在小程序上下文中需要重新定义（建议：单页会话占比 = 整个session仅访问一个页面的用户占比）
- **修复**: 明确小程序的"跳出"定义：session内仅触发一次page_view的用户占比。或标注此指标为"Web端概念，小程序需重新定义"。

#### P1-07: 兑换码激活后用户行为追踪 数据链路未闭环 [数据流断裂]

- **问题**: 方案设计§3.4.5 定义兑换码统计第4项"激活用户后续行为（激活后7日内核心事件完成率）"。
- **偏差**: 当前数据模型中 `user_profiles.activationSource` 和 `activationCodeBatch` 字段标记为新增，但 `user_events` 未定义激活事件与后续事件的关联查询路径。
- **影响**: 无法回答"哪个批次的兑换码带来的用户质量最高"。
- **修复**:
  1. 在 `user_events` 中增加 `activationBatchId` 字段（激活时写入）
  2. 或在查询时通过 `user_profiles.activationCodeBatch` 关联

### 路线图一致性维度

#### P1-08: 方案设计Phase 1耗时12天 vs 需求评审修正后8天 [排期冲突]

- **问题**: 同P0-01的子问题。两份路线图的Phase 1工时差距4天。即使以需求评审修正版为准，方案设计文档也需要更新。
- **修复**: 更新方案设计§九 路线图，采纳需求评审修正版本。

### 安全合规维度

#### P1-09: 开票管理模块缺少税务合规设计 [合规缺口]

- **问题**: 方案设计§3.4.6 开票管理定义了开票流程，但缺少：
  1. 发票号连续性保证（跳号检测）
  2. 发票红冲流程（作废已开发票的处理）
  3. 发票PDF防篡改存储（当前仅存URL）
- **影响**: 税务合规风险。
- **修复**: 补充发票号连续性校验、红冲流程定义、PDF哈希存证。

#### P1-10: daily_stats_snapshots 缺少数据一致性校验 [数据质量问题]

- **问题**: 方案设计未定义快照数据的一致性校验——如果定时任务执行时某些集合的写入正在发生，快照可能捕获到中间态。
- **修复**:
  1. 快照计算使用当日0点前的数据（`createdAt < 当日0点`）
  2. 增加 cross-check：快照中的 `newUsers` 应与 `user_profiles` 当日新增数一致

## 🟢 P2 — 建议 (后续迭代)

### PRD对齐维度

#### P2-01: 运营后台模块命名与小程序模块名不一致

- **问题**: 运营后台模块名"路径分析/财务看板/内容运营/AI质量监控/合规与安全/客服工单/系统健康"与小程序7大模块名"攻略书/证件夹/提醒器/流程控/指引牌(AI)/效率宝/信息权"无映射关系。
- **影响**: 运营人员和产品团队使用两套术语，沟通成本高。
- **建议**: 运营后台增加"对应小程序模块"标注列，或在模块首页增加映射说明。

#### P2-02: 方案设计未定义运营后台的移动端适配策略

- **问题**: §8.3 设计规范写"响应式布局：优先桌面端（1920×1080），兼顾平板"，未提及手机端。实际运营人员可能需要在手机上快速查看核心指标。
- **建议**: Phase 3增加移动端核心指标页面（复用小程序 admin-db 入口）。

#### P2-03: 内容运营"搜索热词Top20"去重逻辑未定义

- **问题**: §3.5 搜索热词定义为"AI对话中的高频问题Top20"。AI对话的query是自然语言，"香港优才计划申请条件"和"优才申请要什么条件"应归为同一热词。
- **建议**: 增加query归一化策略（去停用词 + 关键词提取），或标注Phase 2先人工归类。

#### P2-04: 方案设计缺少运营后台的权限矩阵细节

- **问题**: §3.1 模块总览标注了目标用户角色（PM/运营/内容/客服），但未细化到模块×操作粒度的权限矩阵（如：运营可看财务数据但不可开票，客服可处理工单但不可看收入）。
- **建议**: 补充RBAC权限矩阵表（角色×模块×操作类型）。

### 数据流闭合维度

#### P2-05: admin-revenue 云函数需同时支持微信支付和兑换码两种收入归因

- **问题**: 方案设计§3.4 收入构成含"按付费渠道（微信支付/兑换码激活）"，但 `orders` 集合当前可能仅记录微信支付订单。兑换码激活不产生订单记录。
- **建议**: 兑换码激活时写入虚拟订单（金额=0, channel=redemption_code, planId=对应套餐），便于收入归因统一查询。

#### P2-06: page_view_logs 的 sessionId 生成策略未定义

- **问题**: §3.3.5 页面埋点定义含 `sessionId` 字段用于用户路径桑基图，但未说明sessionId的生成规则（何时创建新session？小程序切后台多久算新session？）。
