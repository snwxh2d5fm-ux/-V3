## 一、总体评价

方案结构清晰，三层命名空间体系设计合理，架构分层（入口/采集/上报/会话/采样）完整。与现有 `utils/tracker.js` 的向后兼容策略可行。

#### 🔴 隐私合规缺失（必须补充）

当前方案使用 `_openid` 作为用户标识，直接存入 CloudBase `user_events` 集合。这构成**个人数据处理**，需要：

1. **用户知情同意** — 首次启动时弹窗告知「我们会采集使用数据以优化服务」，提供 opt-out 开关。隐私政策页需更新。
2. **数据最小化** — `payload` 字段无约束，可能包含 PII。需定义 payload 白名单字段 + 敏感字段脱敏规则（如不记录具体证件号、不记录精确位置）。
3. **保留期限** — 原始事件数据保留多久？建议：详细日志 90 天，聚合数据 永久。需在设计中明确。
4. **K2 合规** — 项目已有 `docs/sop/miniprogram-dev-standards.md` K2 红线。`_openid` 在 CloudBase 数据库中的存储需确认符合 K2 安全规则。
5. **数据删除权** — 用户注销/退出时，需提供删除其埋点数据的入口。

#### ⚠️ 上报机制缺失

- **payload 大小限制**: 小程序 `wx.cloud.callFunction` 单次调用数据上限约 1MB。批量 5 条事件，若单条 payload 含 JSON 大对象，可能超限。需增加 payload 截断策略。
- **失败重试**: 仅提及「离线缓存兜底」，未说明网络恢复后的重放策略（全量重放 vs 增量？去重是否依赖服务端？）
- **云函数不存在**: `usage-tracker` 云函数当前**尚不存在**（项目代码库中搜索不到）。方案中假设已有 `track/stats/userProfile` 三个 action，但实际上整个云函数需要从零搭建。

## 三、必须修改项（放行条件）

### 🔴 致命缺陷（不改不放行）

1. **补充隐私合规专章** — 至少覆盖：用户知情同意机制、数据最小化、保留期限、K2 合规声明
2. **更正 usage-tracker 状态** — 明确该云函数需从零搭建，非「增强」。调整为 P1 优先级
3. **增加 payload 约束** — 定义 payload 白名单字段 + 大小限制 + 敏感字段脱敏规则

### 🟡 建议修改项（建议改，不阻塞放行）

4. **增加测试方案** — 至少：SDK 单元测试（JSDoc类型覆盖）、单页集成测试（page:view自动触发验证）
5. **细化上报器机制** — 补充：payload 截断策略、失败重试策略、服务端去重方案

## 四、技术细节建议

1. **内存队列**: 500 条对低端机型偏大。建议降为 200 条，触发批量上报阈值从 5 条降为 3 条。
2. **去重窗口**: 1 秒太短。快速页面切换（切 Tab）可能触发连续 `page:view`。建议 500ms，以毫秒精度。
3. **生命周期劫持**: `Page()` 包装需注意小程序基础库兼容性。建议测试 2.10.0+ 基础库。
4. **事件映射表**: `assessment_completed → biz:assessment_complete` 这类映射应抽到独立 `mapping.js`，方便维护。
